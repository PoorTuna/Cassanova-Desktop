## v0.1.4

Theme work picked up where 0.1.3 left off. The picker now stays in sync no matter which side flips the switch, and the chrome around the webview stops looking like a different app.

### Theme

- **Two-way sync with Cassanova.** Changing the theme inside an instance's own settings now mirrors back to the desktop chrome. The webview preload watches `localStorage.selectedTheme`, sends the id (including `system`) to the host, and the chrome store dedupes echoes from its own pushes. Previously this was one-way only — the chrome could push, but the webview couldn't.
- **Chrome content area follows the active theme.** The route container, error boundary, and `<body>` were pinned to the shadcn `bg-background` token, which is hardcoded `#08080a` for the dark family regardless of the chosen theme. Switching to Monokai or Solarized left a black halo around the settings card. Fixed by routing those surfaces through the per-theme `bg-cass-app` token.
- **Native window chrome themes too.** A new `window:setChromeColors` channel pushes the theme's background and text colors to the main process on every apply. `BrowserWindow.setBackgroundColor` removes the dark-grey paint flash on theme change; `setTitleBarOverlay` repaints the Windows minimize/maximize/close button strip. macOS uses traffic lights and is skipped via try/catch; Linux is a no-op.

### Build

- Webview preload no longer imports `@shared/ipc-contract`. Two preload entries importing the same shared module made Rollup split it into `out/preload/chunks/`, and sandboxed Electron preloads cannot resolve relative chunk requires — the chrome preload would silently fail and `window.cassanova` came back undefined. The webview-side channel name is inlined as a literal with a comment pointing back to the canonical `IpcChannels` constant.

### Known limitations

- **Cold-start paint flash uses the default colors.** The initial `BrowserWindow.backgroundColor` is still hardcoded; the theme is applied a frame after the renderer mounts. The flash is short but visible on slower machines.
- **Windows 10 taskbar icon renders incorrectly.** The current `build/icon.png` uses straight alpha with translucent edges, which collapses to the legacy 1-bit AND mask on Win10 and leaves only the densely opaque crown visible. Win11 renders the same file correctly. Will be addressed in a follow-up with a hand-tuned `.ico`.
